# Performance Scan — 2026-04-26

Coordinated swarm scan of LogHunter (3 zone analysts + 1 reviewer). Findings are independently verified against source unless flagged otherwise.

## Tier 1 — Web layer (low effort, broad payoff)

### 1. Embedded asset re-read on every request
- `Web/Hosting/WebAppHost.cs:266-295` — `ReadEmbeddedTextBySuffix` does `assembly.GetManifestResourceNames().FirstOrDefault(...)` (LINQ over full manifest) and `reader.ReadToEnd()` per GET, for every CSS/JS asset. No cache, no `ETag`, no `Cache-Control`.
- **Fix:** load each asset once at startup into `Dictionary<string,(byte[] body, string contentType, string etag)>`; serve precomputed bytes; honor `If-None-Match` → 304.

### 2. Sync-over-async on the listener thread
- `WebAppHost.cs:272` — `WriteTextAsync(...).GetAwaiter().GetResult();`
- **Fix:** make `TryServeAsset` `async Task<bool>` and `await` the write.

### 3. Sequential accept loop
- `WebAppHost.cs:53-67` — `await HandleAsync(context)` runs inline; the next `GetContextAsync()` is not issued until handling completes. One slow request stalls every other one.
- The `Task.WhenAny(..., Task.Delay(500))` adds up to 500 ms of artificial latency on a quiet listener.
- **Fix:** dispatch each context via `_ = Task.Run(() => HandleAsync(context))` and re-arm `GetContextAsync()` immediately. Use a `CancellationToken` instead of `Task.Delay`.

### 4. `WriteIndented = true` on hot polling JSON
- `WebAppHost.cs:24`, `Web/Api/AlbApi.cs:21`, `IisApi.cs:19`, `AbuseIpApi.cs:19`, `PlatformApi.cs:18`. Every poll endpoint (1-4 s intervals) ships pretty-printed JSON.
- **Fix:** drop `WriteIndented = true` (or guard behind `?debug=1`); hoist `JsonOptions` to a single shared static; consider `JsonSerializer.SerializeToUtf8Bytes` to skip the intermediate string allocation.

### 5. Other Web findings (medium)
- `WebAppHost.cs:182-205` — `TryDetectViewerKind` opens a SQLite connection on the listener thread.
- API handlers chain four sequential `await TryHandle(...)` calls (`WebAppHost.cs:104-122`).
- `Api/*.cs` `ReadJsonAsync` reads body to string then deserializes (vs `DeserializeAsync(request.InputStream, ...)`).
- JS polls (alb 1 s, iis 1.5-2 s, site 4 s, platform 1.5 s, abuseip 1.8 s) never stop on `state==='completed'`, never pause on `visibilitychange`, and full-rebuild DOM every tick.

## Tier 2 — Service-layer parsing (highest single CPU win)

### 6. Multi-pass tokenization (~10× scans per ALB row)
- `AlbIpSummaryScanner.cs:317-346` and `AlbStatusMismatchScanner.cs:139-166` call `TryGetToken(span, N, ...)` once per token index (1, 3, 4, 5, 6, 7, 8, 9, 12, 13, 22). `TryGetToken` walks from index 0 every call.
- Worst index is 22, so each row is effectively scanned ~10× (reviewer confirmed; analyst said "6+", actual is worse).
- **Fix:** add a single-pass `TokenizeIntoOffsets`-style helper (the IIS reader at `Utils/IisW3cReader.cs:205` already does this) producing `int[]` of (start,len) pairs; index by token number.

### 7. `NormalizeToken` double-allocates per field
- `AlbIpSummaryScanner.cs:486-493`, `AlbStatusMismatchScanner.cs:268-275`, `IisIpSummaryScanner.cs:380-387` — `token.ToString().Trim()` allocates twice when whitespace exists, called 6-10× per row.
- **Fix:** `token = token.Trim()` on the span first; single `ToString()` only when non-empty; share a `"-"` constant.

### 8. Interpolated dictionary keys in hot path
- `AlbScanner.cs:407` `$"{ip}\t{uri}"` — once per parsed row (high volume).
- `AlbStatusMismatchScanner.cs:43` `$"{row.ElbStatusCode}->{row.TargetStatusCode}"` — only on filtered mismatch rows (low volume).
- **Fix:** for status pairs use `Dictionary<(int,int), int>`; for ip+uri use nested `Dictionary<string, Dictionary<string,int>>` so the IP string is interned.

### 9. Other Services findings (medium)
- `IisOption_FindBurstPatterns.cs:344, 356` — per-row `ua.ToString()` and `uri.ToString()` regardless of dedupe.
- `AlbScanner.cs:286` — `fs.Position` read every line for progress.
- Top-N via `OrderByDescending().Take(n)` (`AlbIpSummaryScanner.cs:174`, `AlbStatusMismatchScanner.cs:73`) → use bounded `PriorityQueue`.
- `IisIpSummaryScanner.cs:179-198` maintains two `SortedDictionary<DateTime, ...>` per IP — use `Dictionary` + sort once.
- `AbuseIpDbClient.cs:52` — `HttpClient` per instance; switch to `IHttpClientFactory`.
- `try/catch` around `new DateTime(...)` in `IisIpSummaryScanner.cs:369-377` — validate ranges first.

## Tier 3 — File-level parallelism (debatable win, see "I/O bound" note below)

### 10. No `Parallel.`/`Task.WhenAll`/`Channel.` anywhere in `Services/`
- All scanners loop sequentially: `AlbScanner.cs:261-475`, `IisIpSummaryScanner.cs:253-313`, `IisOption_FindBurstPatterns.cs:251-367`, `AlbStatusMismatchScanner.cs:100-134`, `AlbIpSummaryScanner.cs:231-311`, `AlbOption_RequestsPerIp5Min.cs:101`.
- **Fix:** `Parallel.ForEachAsync(files, new ParallelOptions { MaxDegreeOfParallelism = Environment.ProcessorCount })` with per-file local accumulators merged at end.

## Tier 4 — Viewer (concurrent-UI fix)

### 11. Single shared `SqliteConnection`
- `Viewer/AlbIpSummarySqliteViewerHost.cs:28,40-41,191-192` and `Services/IisIpSummarySqliteViewerHost.cs:25,36-37,157-158`. Concurrent `Promise.all` from the UI (`/api/metadata` + `/api/presets` + `/api/rows`) serializes through one connection.
- **Fix:** open a short-lived ReadOnly connection per request (the file is read-only anyway).

### 12. `LIMIT/OFFSET` deep-paging + `COUNT(*)` per page request
- Same files as #11. With multi-million-row Hits tables, OFFSET-N reads and discards N rows; `COUNT(*)` over filter re-runs every page.
- **Fix:** seek pagination (`WHERE TimestampUtc < $lastTs OR (... AND Id < $lastId)`); cache filtered count per filter-hash for the request lifecycle.

### 13. CSV export blocks the entire viewer
- `AlbIpSummarySqliteViewerHost.cs:247-300`, `IisIpSummarySqliteViewerHost.cs:189-226` — unbounded `SELECT` over the only connection while the listener is also single-threaded (per Tier 1 #3).
- **Fix:** dedicated short-lived connection + async reader.

### 14. `Task.Delay(500)` polling
- `AlbIpSummarySqliteViewerHost.cs:64-66` and IIS twin. Wakes 2×/s even when idle.
- **Fix:** `await getContextTask.WaitAsync(cts.Token)`.

### 15. Read-side PRAGMAs
- Writers DO create indexes and apply `journal_mode/synchronous/temp_store/locking_mode` (`Services/AlbIpSummaryExportSqlite.cs:139-144`, `Services/IisIpSummaryExportSqlite.cs:149-153`) — analyst's "no indexes" claim was REFUTED.
- However, viewer hosts don't apply read-side `PRAGMA mmap_size=268435456; PRAGMA cache_size=-200000;` at open time. Worth adding.

## Refuted findings (do not act on)

- "Burst scanner UA `ToString()` before filter" (`IisOption_FindBurstPatterns.cs:290`) — REFUTED. Filter already runs first.
- "Viewer DBs have no indexes" — REFUTED. Writer side creates them.

## Recommended landing order

1. **Tier 1 — Web caching + un-indent JSON + parallel accept loop** (~1 day; helps every page).
2. **Tier 2 — Single-pass ALB tokenization + `NormalizeToken` cleanup** (~1 day; 5-10× CPU win on hot ALB workflows).
3. **Tier 4 — Viewer connection-per-request + seek pagination** (~half day; unblocks concurrent UI).
4. **Tier 3 — `Parallel.ForEachAsync` over files** (~half day; payoff depends on workload — see I/O analysis).
